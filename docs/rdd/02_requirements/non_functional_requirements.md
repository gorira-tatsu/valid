# 03. Non-Functional Requirements

- ドキュメントID: `RDD-0001-03`
- バージョン: `v0.3`
- 目的: 本システムの品質特性を、実装・運用・監査まで含めて定義する。

## 1. 本章の位置づけ

本章は「何ができるか」ではなく、「どの品質でできなければならないか」を定義する。形式検証基盤では、機能要件を満たしていても、判定が不安定、再現不能、監査不能、遅すぎる、または誤PASSを返す設計であれば業務投入に耐えない。したがって、本章は実装の後から付け足す品質メモではなく、設計そのものを拘束する上位仕様として扱う。

本システムの非機能要件は、一般的なCLIツールより厳しい。理由は3つある。第一に、AIが主要利用者であるため、曖昧な失敗や人間依存の診断文では運用できない。第二に、CIゲートとして使うため、再現性と決定性が必須である。第三に、最終的にはセルフホストを目標にしているため、信頼境界と監査線が曖昧では成立しない。

## 2. 品質属性の全体方針

本システムでは、品質属性を以下の順序で優先する。

1. 正確性
2. 再現性
3. 監査性
4. 安全性
5. 運用性
6. 性能
7. 拡張性
8. 可搬性

この順序は意図的である。たとえば性能のために判定意味論が変わることは許容しない。高速だが信頼できない結果より、遅くても証拠付きで再現可能な結果を優先する。

## 3. 正確性

### 3.1 最上位原則

- `PASS` は最も重い主張であり、誤って返してはならない。
- `FAIL` は必ず証拠トレースを伴う。
- `UNKNOWN` は成功扱いしない。
- `ERROR` は内部不整合または環境不備として、`PASS/FAIL/UNKNOWN` と分離する。

### 3.2 判定意味論の固定

非機能要件として、判定意味論を文章で固定する。

- `PASS`: 指定した探索条件と意味論の範囲で、対象プロパティが成立した。
- `FAIL`: 1つ以上の証拠トレースによりプロパティ不成立が観測された。
- `UNKNOWN`: 上限到達、未対応構文、外部ソルバ制約などにより、成立・不成立のいずれも断定できなかった。
- `ERROR`: パーサ異常、型検査異常、内部例外、入出力障害、契約不整合など、検証行為そのものが成立していない。

### 3.3 誤判定防止策

- 明示探索とBMCのいずれでも、最終出力は共通trace形式へ正規化する。
- `FAIL` はカーネルによるtrace再生確認を経て確定する。
- `PASS` はその時点の前提条件、ソルバ能力、探索制限を必ず結果に同梱する。
- `UNKNOWN` の理由はカテゴリ別に構造化出力する。

### 3.4 正確性SLO

- 反例trace再生一致率: 100%
- `FAIL` でtrace欠落率: 0%
- 互換スキーマ上での結果パース失敗率: 0.1%未満

## 4. 再現性

### 4.1 再現性の定義

同一の入力モデル、同一のソルバ能力宣言、同一の実行設定、同一のseed、同一のバージョン集合のもとで、同一のステータスと同一の証拠が得られることを再現性と定義する。

### 4.2 保存必須情報

再現性のため、以下をすべて保存する。

- `source_hash`
- `schema_version`
- `engine_version`
- `backend_name`
- `backend_version`
- `limits`
- `seed`
- `platform_metadata`
- `trace_hash`
- `contract_hash`

### 4.3 再現不能時の扱い

再現不能は軽微障害ではない。以下の順序で扱う。

1. `trace_replay_mismatch`
2. `solver_output_instability`
3. `environmental_drift`
4. `schema_drift`

いずれもインシデントとして記録し、同一原因の再発を防止する。

### 4.4 再現性目標

- 同一seed再現失敗率: 1%未満
- 同一trace再生不一致率: 0%
- 契約ハッシュ誤差検出漏れ: 0%

## 5. 監査性

### 5.1 監査対象

本システムでは以下を監査対象とする。

- モデル定義
- 実行設定
- 検証結果
- 証拠トレース
- 生成テスト
- 契約差分
- ドキュメント生成差分

### 5.2 監査証跡

監査証跡は人間が読むログではなく、構造化データとして保存する。最小単位は `run_id` であり、`run_id` から以下を辿れる必要がある。

- 対象モデル
- 対象プロパティ
- 利用バックエンド
- 実行パラメータ
- 結果
- 生成された証拠
- その後の最小化処理
- 生成されたテスト

### 5.3 監査設計要件

- 証拠のハッシュを保持する。
- 派生物は元runへ逆参照できる。
- `UNKNOWN` と `ERROR` も監査対象に含める。
- AIによる修正提案には根拠trace IDを付与する。

## 6. 安全性

### 6.1 入力安全性

- パース失敗でpanicしない。
- 巨大入力で無制限メモリ消費しない。
- MermaidやJSON出力に危険文字列をそのまま埋め込まない。

### 6.2 実行安全性

- 外部ソルバは直接シェル文字列連結で起動しない。
- 一時ファイルは衝突しないディレクトリへ隔離する。
- 実行上限を超えた場合は強制中断可能である。

### 6.3 生成コード安全性

- testgen結果は安全なエスケープを行う。
- 生成コードが追加で任意コード実行経路を作らないこと。
- `include!` を使う場合も固定出力先とハッシュ管理を行う。

## 7. 性能

### 7.1 基本方針

性能は「速いこと」よりも「制御できること」を重視する。状態空間爆発は避けられないため、速度目標に加え、停止性と説明可能性を品質要件として持つ。

### 7.2 MVP性能要件

- 明示探索で `10^5` 状態は開発機で現実的時間内に処理できる。
- 適切なハッシュ戦略により `10^6` 状態まで段階的に対応可能である。
- 反例復元コストは探索コストの範囲内に抑える。

### 7.3 性能計測項目

- 探索状態数
- 遷移試行数
- 有効遷移率
- 枝刈り率
- 訪問済み率
- 反例復元時間
- coverage集計時間
- メモリ使用量

### 7.4 性能SLO

- 主要ユースケースでPR時検証は10分以内を目標。
- nightly深掘り検証は1時間以内を目標。
- coverage集計は検証時間の20%以内を目安。

## 8. 可用性

### 8.1 CLI可用性

- 失敗時でも意味のある終了コードを返す。
- 標準出力と標準エラーの責務を分離する。
- `--json` 指定時は機械可読出力を壊さない。

### 8.2 API可用性

- スキーマバージョンを必須化。
- 不完全応答を返さない。
- 失敗分類は常にコード化する。

### 8.3 CI可用性

- 環境差異で判定が変わる場合は明示警告。
- optional backend不在時の振る舞いを定義する。
- 必須backend不在時は即失敗する。

## 9. 保守性

### 9.1 コード構造

- kernelを最小化する。
- solverごとの複雑性をadapterへ閉じ込める。
- parser/typechecker/evaluator/reporter/testgenを分離する。

### 9.2 保守ルール

- `unsafe` はkernelで禁止。
- public JSON schema変更には移行ノート必須。
- deprecated optionには期限付き廃止計画を付ける。

### 9.3 保守性指標

- kernel行数
- adapter追加時の既存変更行数
- schema migration件数
- backward compatibilityテスト成功率

## 10. 拡張性

### 10.1 拡張対象

- 新しいソルバ
- 新しいproperty種別
- 新しい描画形式
- 新しいAI API
- 新しいカバレッジ指標

### 10.2 拡張制約

- 既存trace readerを壊さない。
- core domain語彙を増やしすぎない。
- backend固有概念をdomain語彙へ昇格させる際はADR必須。

## 11. 互換性

### 11.1 スキーマ互換

- `schema_version` をメジャー/マイナーで管理。
- キー追加は後方互換として扱う。
- 意味変更や削除はメジャー変更とする。

### 11.2 CLI互換

- フラグ名は安易に変更しない。
- 既存フラグ廃止は2リリース以上の猶予を設ける。

### 11.3 trace互換

- 旧trace readerを可能な限り維持。
- 互換を切る場合は変換ツールを同時提供。

## 12. 可搬性

### 12.1 対応環境

- Linux
- macOS
- Windows

### 12.2 依存制約

- 標準的なRustツールチェインでビルド可能。
- 外部ソルバ依存はfeatureまたはadapterで分離。
- OS依存パス操作をdomain層へ漏らさない。

## 13. DDDから見た非機能要件

DDD観点では、非機能要件は単なる運用メモではなくDomain Policyとして扱う。代表例は以下である。

- `UnknownPolicy`
- `TraceRetentionPolicy`
- `CoverageGatePolicy`
- `ContractChangePolicy`
- `AuditRetentionPolicy`

これらはUse Caseが参照する政策であり、ControllerやCLIが勝手に判断してはならない。

## 14. クリーンアーキテクチャから見た非機能要件

### 14.1 Entities

Entitiesは品質要件を直接知らないが、品質判断に必要なメタデータを保持する。たとえばTraceは `hash` を、Runは `limits` を保持する。

### 14.2 Use Cases

Use Casesが品質制約を執行する。例として `CheckUseCase` は time limit、state limit、unknown handling policy を適用する。

### 14.3 Interface Adapters

AdaptersはDomain Errorを外部表現へマッピングする。性能や可用性のためにDomain意味論を変えてはならない。

### 14.4 Framework/Drivers

外部ソルバ、FS、CI環境は最も不安定な層であり、非機能リスクの主因である。したがって、この層の異常は必ず構造化して上位へ伝える。

## 15. STO/SSOTから見た非機能要件

STO違反は品質劣化そのものである。一次ソースと派生物が食い違う状態を許すと、再現性・監査性・正確性が同時に崩れる。そのため、以下を必須とする。

- 一次ソースに `source_hash` を付与する。
- 生成物は `source_hash` を保持する。
- `doc --check` と `contract --check` をCIの必須ゲートにする。
- 派生物の手編集は禁止または失敗扱いとする。

## 16. ソルバから見た非機能要件

### 16.1 ソルバ中立性

非機能要件はソルバごとの差異を吸収した上で上位へ見せる必要がある。たとえばsolver Aが `timeout`、solver Bが `resource exhausted` を返しても、上位では `UNKNOWN_LIMIT` などの共通コードへ正規化する。

### 16.2 ソルバ能力宣言

各ソルバadapterは能力を自己宣言する。

- supports_invariant
- supports_reachability
- supports_deadlock
- supports_liveness
- supports_witness
- supports_counterexample
- supports_unsat_core

### 16.3 ソルバ評価軸

- 決定性
- trace変換容易性
- 診断の質
- 実行時間
- メモリ特性
- 導入難度

## 17. データモデルから見た非機能要件

非機能要件はデータスキーマに直接現れる。最低限、以下が必要である。

- Runにlimitsとbackend情報
- Traceにhashとkind
- CoverageReportに集計条件
- ContractSnapshotにhashとlock version
- すべてにschema version

これがないと比較、監査、再現ができない。

## 18. AI運用から見た非機能要件

AIにとって最大の品質要求は「曖昧さがないこと」である。したがって以下を要求する。

- 失敗コードは固定列挙。
- explainは構造化配列で返す。
- 人間向け長文ログのみで重要情報を伝えない。
- 途中失敗でもJSON外形を維持する。

## 19. 品質ゲート

### 19.1 必須ゲート

- contract mismatchなし
- FAIL without traceなし
- schema invalidなし
- source/generated driftなし

### 19.2 条件付きゲート

- coverage threshold
- UNKNOWN threshold
- performance regression threshold

### 19.3 例外規定

緊急障害対応では条件付きゲートを一時緩和できるが、必須ゲートは緩和しない。

## 20. 監視・可観測性

### 20.3 アラート

## 21. リスクと品質観点の緩和

### 21.1 状態爆発

性能問題に見えるが、結果的にはUNKNOWN多発による可用性問題でもある。対策としてlimit理由の可視化、分割提案、抽象化提案を行う。

### 21.2 モデル誤り

正確性の外部要因である。対策はモデルレビュー手順、REQ-ID追跡、explainの改善。

### 21.3 ソルバ不整合

trace正規化とkernel再生で吸収する。

### 21.4 AI暴走

contract/doc/checkを通らない変更を自動反映しない。

## 22. 章固有の受け入れ基準

- 本章だけで品質属性の優先順位が理解できる。
- 各属性に測定方法または判断基準がある。
- DDD/CA/STO/ソルバ観点が独立節として定義されている。
- 実装チームがSLOとゲート条件をそのままCIへ落とし込める。

## 23. 章固有の文書管理規約

- 本章に機能要件を持ち込まない。
- 品質属性ごとに用語を固定する。
- 新しい品質属性を追加する場合、優先順位と測定法を必ず定義する。

## 24. 結論

本章の要点は、非機能要件を「あとで頑張る」対象にしないことである。形式検証基盤においては、品質特性そのものが製品価値であり、正確性・再現性・監査性のいずれかが欠けた時点で基盤は信頼を失う。したがって、本章は実装判断よりも優先して守るべき制約として扱う。

## 25. 品質属性ごとの運用SOP

### 25.1 正確性SOP

- 新しいbackendを追加した際は、既知のgolden traceセットで再生一致試験を行う。
- `PASS` 判定を返す変更は、最低1つの回帰シナリオで以前の結果と比較する。
- `FAIL` にtraceが含まれない場合は、その実行を無効結果として扱う。

### 25.2 再現性SOP

- release前にseed固定の再実行試験を行う。
- 同一入力で3回以上結果が揺れる場合は、非決定要因の調査が完了するまで本番ゲートへ昇格させない。

### 25.3 監査性SOP

- run_id単位のアーティファクト欠落があれば失敗扱いにする。
- 監査ログの保存期限は、少なくとも主要バージョン2世代分を維持する。

## 26. 性能予算

本システムの性能は無限に改善対象にしない。性能予算を定義して、その範囲内で設計する。

- parser/typecheck: PR時1分未満
- explicit check: PR時10分未満
- testgen: 単一traceあたり1分未満
- coverage: 主要モデル群で5分未満
- contract/doc drift check: 1分未満

性能予算を超える場合、まず設計のどこで予算を使っているかを可視化する。闇雲な高速化は許容しない。

## 27. 可用性の詳細設計

### 27.1 Degraded Mode

一部backendが利用不能でも、システム全体が停止しない設計が望ましい。たとえばBMC backendが未導入ならexplicitだけで稼働し、その事実を応答へ明示する。これはフェイルオープンではなく、能力低下を明示したうえでの限定運用である。

### 27.2 Hard Fail条件

以下は即失敗とする。

- schema_version不一致で読めないtrace
- contract hashの読み取り不能
- kernel replay失敗
- source_hash未設定

## 28. セキュリティの詳細設計

### 28.1 Artifact Hygiene

生成されるartifactsは、相対パス汚染や上位ディレクトリ脱出を起こさない命名と配置を採用する。`run_id` や `evidence_id` をファイル名へ使う際も、英数字と安全な区切り文字に制限する。

### 28.2 Confidentiality

本システムは主に仕様と証拠を扱うが、検証対象によってはドメイン上の機密情報がstateへ含まれる可能性がある。そのため、ログ・trace・explainはマスキング戦略を持つべきである。少なくとも、公開CIへ出すartifactと内部監査用artifactを分ける余地を残す。

## 29. DDD品質ポリシーの具体例

### 29.1 UnknownPolicy

- `strict`: UNKNOWNは常に失敗
- `warn`: UNKNOWNは警告
- `budgeted`: UNKNOWN件数や比率が閾値以内なら許可

### 29.2 CoverageGatePolicy

- `transition_min`
- `guard_min`
- `critical_property_must_cover`

### 29.3 AuditRetentionPolicy

- `short`: 30日
- `standard`: 180日
- `long`: 365日以上

これらは設定値ではあるが、Domain Policyとして扱うことで、CLIやCIが個別判断を持たずに済む。

## 30. クリーンアーキテクチャ上の品質責務

品質要件は層ごとに責務を持つ。

- Entities: 整合性に必要なメタデータを保持する
- Use Cases: 上限、policy、判定ルールを適用する
- Adapters: 安定した出力契約を維持する
- Drivers: 外部不安定性を隔離する

この責務分割を曖昧にすると、たとえばCLIがUNKNOWNを握りつぶす、solver adapterが勝手にtraceを省略する、などの品質劣化が起こる。

## 31. 品質回帰テスト戦略

非機能要件そのものも回帰試験の対象とする。

- golden JSON schema test
- replay consistency test
- contract drift test
- deterministic ordering test
- artifact completeness test

これにより、機能追加の陰で再現性や監査性が壊れることを防ぐ。

## 32. 章固有の補足結論

本章を満たすということは、単に「速くて安定している」ではない。証拠を失わず、意味論を変えず、環境差異を説明でき、AIとCIが安全に扱えることまで含めて品質とみなす。したがって、非機能要件は将来の最適化で緩める対象ではなく、設計の土台そのものである。

## 33. 品質レビューの実施方法

非機能要件は、コードレビュー時に「雰囲気で確認」してはならない。最低限、以下の観点でレビューする。

- この変更はtrace再現性へ影響するか。
- schema互換性を壊していないか。
- contract/doc/checkのゲート条件を緩めていないか。
- backend依存の意味差を隠していないか。
- AIが読むJSONの安定性を壊していないか。

## 34. 性能劣化の取り扱い

性能は重要だが、単純に遅いことと、停止しないことは区別する。停止性を失う変更は重大欠陥、数十%の性能劣化は要調査とする。さらに、性能劣化は「どの層で起きたか」を特定する必要がある。frontend、engine、reporter、artifact保存で対策は異なるためである。

## 35. 非機能要件の計測基盤

計測を別システムへ委ねると品質が形骸化するため、最低限の計測機能を本体へ持たせる。

- 実行時間計測
- replay成功率
- JSON schema validation成功率
- artifact生成完了率
- backend availability

これにより、非機能要件を人手報告ではなく実データで運用できる。

## 36. 将来拡張に対する非機能制約

新しいsolverや新しいpropertyを追加する時、最初に確認すべきは機能適合ではなく、非機能適合である。

- traceを共通形式へ落とせるか
- deterministic modeを持てるか
- timeoutやlimit理由を構造化できるか
- CIで現実的に回せるか

この条件を満たさない技術は、研究価値が高くても本番ゲートへは採用しない。

## 37. 最終補足
