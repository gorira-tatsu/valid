# 01. Overview and Scope

- ドキュメントID: `RDD-0001-01`
- バージョン: `v0.2`
- 目的: 本プロジェクトの全体像、境界、設計思想、開発原則、運用原則を定義する。

## 1. プロジェクト定義

本プロジェクトは、Rust記法を中心とした形式検証基盤を構築し、仕様と実装の乖離を縮小しつつ、反例・証拠を継続的にテスト資産化することを目標とする。主なユーザーは人間開発者だけではなくAIエージェントを含む。したがって、機械可読性、決定性、再現性が一次要件となる。

本プロジェクトのアウトプットは単体の検証器ではない。モデル記述、検証実行、反例最小化、テスト生成、カバレッジ計測、CI連携、契約管理、ドキュメント生成を一連で扱える「検証運用プラットフォーム」である。

## 2. 何を解くか

### 2.1 直接解く課題
- 仕様レビューと実装レビューが分離し、整合維持コストが高い。
- 並行遷移・境界値・稀条件のバグがレビューで取りこぼされる。
- 形式検証結果が現場の回帰テスト資産にならない。
- AIが安全に検証を自動実行できる安定I/Fがない。

### 2.2 期待する変化
- 人間: モデル・要件の妥当性に集中する。
- AI: 検証の反復実行、反例の短縮、テスト生成、説明生成を担当する。
- CI: 品質ゲートとして証拠ベースの判定を行う。

## 3. スコープ境界

### 3.1 In Scope
- Rust trait/macroによるモデル埋め込み。
- 明示状態探索（MVP）。
- 反例トレース復元と可視化。
- 反例および満たす例のテストベクタ化。
- 契約ハッシュとlock管理。
- AI向けJSON API。

### 3.2 Out of Scope（初期）
- Rust全構文の完全意味論。
- 完全定理証明を初期コアにすること。
- 単一ソルバ依存での閉鎖的設計。

## 4. 設計原則

### 4.1 Correctness First
- 誤PASSを最も重い欠陥として扱う。
- 不明は不明として返す（UNKNOWN隠蔽禁止）。
- FAILは証拠トレースを必須化する。

### 4.2 Evidence-Driven
- 判定結果だけでなく、証拠（trace, vector, coverage）を保存する。
- 反例は必ず再生可能であること。

### 4.3 SSOT（STO）
本書でのSTOは Single Source of Truth と定義する。仕様、契約、生成物の整合を壊さないために、一次情報を限定する。

- 優先一次ソース: モデル定義（Rust macroまたは仕様ブロック）。
- 生成物: Markdown仕様書、Mermaid、テストベクタ、契約スナップショット。
- ルール: 生成物の手編集を禁止またはCIで検出。

### 4.4 Extensible Solvers
- コア意味論は統一IRで保持し、ソルバは差し替え可能にする。
- 明示探索、SAT/BMC、将来のSMT/演繹を同じ実行計画で扱う。

## 5. DDD方針

### 5.1 Bounded Context
- `Modeling Context`: 要件、モデル、プロパティ記述。
- `Verification Context`: 実行計画、探索、証拠生成。
- `Evidence Context`: trace/vector/reportの保管と照合。
- `Integration Context`: CI、外部ソルバ、AI API。

### 5.2 Ubiquitous Language
- Requirement, Model, Property, Run, Evidence, Vector, Contract, Coverage, Unknown。
- 用語の意味をコード・CLI・ドキュメントで一致させる。

### 5.3 Aggregate設計
- `ModelAggregate`: State/Action/Propertyの一貫性境界。
- `RunAggregate`: 実行設定と結果の整合境界。
- `EvidenceAggregate`: 証拠トレースと派生テストの整合境界。

### 5.4 Domain Events
- `ModelDefined`
- `VerificationStarted`
- `PropertyFailed`
- `EvidenceMinimized`
- `RegressionTestGenerated`
- `ContractChanged`

## 6. クリーンアーキテクチャ方針

### 6.1 層定義
- Entities: State, Action, Property, Trace, Contract。
- Use Cases: Check, Minimize, TestGen, Coverage, ContractCheck。
- Interface Adapters: CLI, JSON API, Mermaid Renderer, Solver Adapter。
- Framework/Drivers: SAT solver process, filesystem, CI runtime。

### 6.2 依存ルール
- 内側は外側を知らない。
- ソルバ実装はAdapterに閉じ込める。
- kernelはI/Oを持たない。

### 6.3 変更耐性
- DSL変更はfrontendとadapterに限定し、kernelを保護。
- ソルバ追加は新Adapterだけで可能にする。

## 7. ソルバ戦略（全体像）

### 7.1 フェーズ戦略
- Phase 1: Explicit（BFS/DFS）で確実に動かす。
- Phase 2: BMC（SAT/SMT）で反例・満たす例の生成効率を上げる。
- Phase 3: 必要箇所で演繹系/記号実行を統合。

### 7.2 ソルバ選定基準
- 決定性
- 失敗時診断の質
- 速度
- CI適合性
- ライセンスと保守性

### 7.3 証拠中心設計
- ソルバ結果は必ず共通trace形式に正規化。
- kernelがtraceを再生し、判定を再確認する。

## 13. 補遺: 設計選択の理由

Rustに寄せる理由は、型・所有権・コンパイラ資産を検証基盤で再利用できるからである。外部仕様言語だけで設計すると、仕様と実装の整合性に常に架橋コストが発生する。trait/macroで同一言語に閉じることで、変更検知、差分レビュー、CI連携が簡潔になる。

Alloy/TLA+をそのまま統合実装するのではなく、統一IRで意味を合流させる理由は、運用継続性にある。ソルバは市場と研究で進化するため、固定化すると陳腐化が早い。IR中心なら、新しいバックエンドを追加しても資産を守れる。

また、AI運用時は、人間向けの曖昧出力が最大の障害になる。例外文言の揺れ、キー名の揺れ、失敗分類の曖昧さは自動化率を下げる。最初から機械可読を主目的に設計することで、AIによる大規模反復が実現できる。

本章は、個別機能の仕様書ではなく、意思決定の一貫性を維持するためのガバナンス文書である。以降の章は本章の原則を具体仕様に落とし込む。

## 14. 補遺: DDD運用規約

- 仕様議論で使う語はUbiquitous Languageに限定する。
- Contextを横断する変更はArchitecture Decision Record（ADR）を必須化する。
- REQ-IDはModeling Contextで発行し、Verification Contextへ参照として渡す。
- Evidence Contextは不変ログとして扱い、後追い監査を可能にする。

## 15. 補遺: クリーンアーキテクチャ運用規約

- Use Case層にソルバ依存コードを置かない。
- Adapter層は例外をDomain Errorへマッピングする。
- Presenter/ReporterはDomain Modelを破壊しない。
- Driver層の変更でEntitiesに変更が必要な場合、設計破綻としてレビューする。

## 16. 補遺: STO/SSOT運用規約

- 仕様の一次ソースは1つに固定する。
- 仕様書を一次ソースにする場合、モデルを埋め込みブロックで抽出可能にする。
- モデルを一次ソースにする場合、仕様書は生成する。
- 一次ソース変更時は必ず契約ハッシュ更新とドキュメント再生成を行う。

## 17. 補遺: ソルバ統合ポリシー

- 新規ソルバ統合はAdapter契約に従う。
- 必須出力: status, trace(optional), stats, diagnostics。
- trace形式に落とせないソルバは本番ゲート対象にしない。
- 実験ソルバは`experimental`フラグで隔離する。


## 18. 詳細方針: モデルレビュー中心の開発プロセス

### 18.1 レビュー単位
- コードレビュー単位ではなく、`ModelDelta`単位でレビューを行う。
- `ModelDelta`はState差分、Action差分、Property差分、Constraint差分で構成する。
- レビューコメントは必ずREQ-IDまたはProperty-IDに紐づける。

### 18.2 レビュー観点チェックリスト
- 仕様の意図: 新規要件は何を保証したいのか。
- 到達性: 追加遷移は到達不能になっていないか。
- 安全性: 既存不変条件を破壊していないか。
- 監査性: 反例が出たときに説明可能か。
- 運用性: CI時間予算を超えないか。

### 18.3 レビューアウトカム
- 承認。
- 条件付き承認（カバレッジ改善要）。
- 差し戻し（REQ未整備、プロパティ未定義、契約差分未管理）。

## 19. 詳細方針: AI運用設計

### 19.1 AI実行モード
- `assistant-mode`: 提案のみ。
- `autofix-mode`: 修正提案+パッチ生成。
- `automerge-mode`: 原則禁止。将来も限定条件。

### 19.2 AI実行制限
- 仕様一次ソースの直接改変は`assistant-mode`では禁止。
- 契約ハッシュ差分を伴う変更は人間承認必須。
- UNKNOWNをPASSとする提案は禁止。

### 19.3 AIの説明責任
- すべての提案に根拠トレースIDを付与。
- 変更提案に期待効果（coverage改善、UNKNOWN削減）を添付。
- 失敗提案の履歴を残し、誤学習を回避。

## 20. 詳細方針: 境界設定

### 20.1 ビジネス境界
- 本基盤は「検証基盤」であり業務機能本体は対象外。
- ただし業務ルールをModelとして受け取る責務を持つ。

### 20.2 技術境界
- 実行時監視ツールではなく、設計時/CI時検証が主責務。
- Runtime assertは補助機能。

### 20.3 組織境界
- 仕様責任はドメインオーナー。
- 実装責任は開発チーム。
- 判定責任はCIポリシーに基づく。

## 22. 失敗パターンと予防

### 22.1 想定失敗
- モデルが過剰抽象で実装乖離。
- テスト生成が多すぎて保守不能。
- ソルバ依存で再現不能。

### 22.2 予防策
- 観測可能状態ベースのオラクル設計。
- テスト生成上限と優先順位制御。
- 共通trace再生による再現性担保。

## 23. 組織導入プラン

### 23.1 導入ステップ
1. 対象領域を1つ選ぶ（状態機械性が高い領域）。
2. REQ-ID体系を整備する。
3. 最小モデルを作る。
4. CIにcheckとcontractを追加。
5. FAILの回帰テスト化を定着させる。

### 23.2 導入基準
- プロパティが明文化できる。
- 状態が有限化できる。
- チームがモデルレビューを受け入れる。

### 23.3 撤退基準
- UNKNOWN比率が高止まりし、対策効果がない。
- モデル作成コストが効果を恒常的に上回る。

## 25. 用語辞書（抜粋）

- Action: 状態遷移を引き起こす操作。
- Property: 満たすべき形式的命題。
- Evidence: 判定の根拠データ。

## 26. 追加設計メモ: 段階的厳密化

初期は有限状態に限定するが、厳密化は段階投入する。

- Step A: bounded intとenumのみ。
- Step B: 有限集合と関係。
- Step C: 時相拡張。
- Step D: 公平性制約。
- Step E: 演繹証明連携。

この段階戦略により、開発コストと価値発現のバランスを取る。

## 27. 追加設計メモ: 監査設計

- runごとに実行設定を固定保存。
- ソルババージョンを記録。
- traceハッシュ、vectorハッシュを記録。
- 監査時は同条件で再実行して一致を確認。

## 28. 追加設計メモ: 技術的負債管理

- 仕様負債（REQ未対応Property）
- モデル負債（到達不能Action）
- 検証負債（UNKNOWN常習）
- 運用負債（手動運用残存）

各負債はIssueタイプを分離して可視化する。

## 29. 追加設計メモ: ガバナンス

- `Architecture Board` を設置し、Context境界変更を審査。
- `Model Owner` を各モデルに割り当てる。
- `Verification Steward` がCI閾値を維持。

## 30. 結論

本章は全章の上位拘束条件である。仕様変更、アーキテクチャ変更、ソルバ変更、AI運用変更は本章の原則に整合しなければ採用しない。特にSTO、DDD境界、クリーンアーキテクチャ依存方向、証拠中心運用は不変方針として扱う。

## 39. 追加補遺: 実運用の判断基準

- 反例が再現できない場合、検証器バグまたは環境差異の疑いとして最優先調査。
- UNKNOWNが連続する場合、モデル分割または抽象化導入を検討。
- テスト生成が過剰な場合、目的関数（coverage target）を再設計。
- 仕様と実装が乖離する場合、一次ソースを再確認し、STO違反を解消。

## 43. 最終補遺

本章は方針章であり、実装詳細は各章へ委譲する。ただし以下のみ固定ルールとして扱う。

- SSOT/STOは常に1つ。
